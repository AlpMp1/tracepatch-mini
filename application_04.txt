我正在构建 TracePatch Studio，一个面向开源项目与小团队的 AI CI 失败诊断与自动修复工作台，目标是解决开发者在维护中大型代码仓库时日志太长、定位太慢、修复成本高、测试回归困难的问题。很多项目在长期迭代后，会积累复杂的依赖关系、异步状态逻辑与历史兼容代码。当框架升级、依赖变更或新增功能时，经常会出现 CI 测试失败、类型错误、缓存失效、状态同步异常等问题。开发者需要阅读大量测试日志、对比 PR diff、分析调用链，再反复修改和运行测试，整个过程耗时且容易遗漏边界条件。TracePatch Studio 希望把这套流程做成一个可复用的 AI Coding 工作流，让模型能够从仓库级上下文中完成问题定位、补丁生成、测试补齐与风险审查。项目的核心由多个 Agent 协同完成。Architect Agent 负责读取仓库目录、package 配置、测试日志、历史修复记录与调用链，生成失败传播路径与依赖影响图；Planner Agent 根据失败类型生成最小风险 patch 方案，并规划回归测试与验证步骤；Patch Agent 在 OpenClaw、OpenCode 等 AI 编程工具中调用 MiMo 生成代码补丁、修复类型错误、补充测试，并根据 lint、typecheck 与 unit test 输出进行多轮修复；Review Agent 对生成结果进行二次审查，输出风险等级、受影响模块、回滚建议与 PR 摘要。最终形成一个可提交、可复盘、可继续迭代的工程化修复结果，而不是普通聊天记录。TracePatch Studio 高度依赖长上下文、多轮推理与 Agent 协作。单次 CI 失败分析需要同时读取测试日志、依赖变更、调用链、失败快照与历史 patch；单个复杂修复任务通常会经历日志分析、文件定位、补丁规划、代码生成、运行测试、分析失败、多轮修复、风险审查与生成报告等多个回合。我计划使用 20-30 个真实开源仓库做 benchmark，覆盖 React、Next.js、Node.js 与 Python 等不同技术栈，验证 MiMo 在 CI 日志理解、依赖升级修复、回归测试生成与 PR Review 中的效果。由于需要持续保留完整上下文、重复执行 patch replay 与多轮测试，因此希望申请较高 Token Plan 额度，用于长期实验与迭代。
